# Guided Administrative Assistant

The Guided Administrative Assistant is an additive workflow layer for SchoolDesk. It does not replace the existing ERP modules, screens, or CRUD paths. Existing Principal/Admin screens continue to own their normal operations; the assistant stores draft workflow state, validates connected dependencies, shows a review summary, and only executes after explicit confirmation.

## Architecture

Flow:

`Assistant UI -> Workflow session API -> Validation/orchestration -> Existing ERP tables/services -> Audit/log records`

The UI is chat-first: the assistant asks one operational question at a time, shows quick reply chips and compact answer cards inside the conversation, and then advances through the configured workflow steps. This is only the presentation layer. Workflow state, validations, suggestions, review, and execution continue to use the deterministic workflow engine.

Backend additions:

- `workflow_sessions`: current workflow, status, current step, completed/pending steps, draft data, validation summary, review summary, user, timestamps, rollback state.
- `workflow_logs`: draft, validation, import, execution, cancellation, and rollback events.
- Existing `audit_logs`: execution is also bridged into the existing audit log system.

Routes:

- `GET /api/v1/assistant/workflows`
- `POST /api/v1/assistant/intent`
- `GET /api/v1/assistant/sessions`
- `POST /api/v1/assistant/sessions`
- `PUT /api/v1/assistant/sessions/:id/steps/:step_id`
- `POST /api/v1/assistant/sessions/:id/validate`
- `POST /api/v1/assistant/sessions/:id/execute`
- `DELETE /api/v1/assistant/sessions/:id`
- `GET /api/v1/assistant/templates/:workflow_type`
- `POST /api/v1/assistant/sessions/:id/import-preview`

All assistant routes are protected by auth, school scope, and Admin/Principal RBAC.

## Current Workflows

- Create Class: class details, sections, subjects, class teacher, subject teachers, fee structure, timetable, settings, review, execution.
- Student Onboarding: student details, guardian details, optional fee invoice assignment, review, execution.
- Teacher Onboarding: staff details, optional login, subject/class mapping, review, execution.
- Fee Setup: academic year/class scope, fee items, review, execution.
- Timetable Setup: section/year/term scope, generation rules, review, execution.

Other action cards such as Attendance, Exams, Notifications, Reports, Transport, Library, and Inventory are present as quick assistant dashboard actions. The currently implemented ones route to stable ERP modules when no dedicated workflow exists yet.

## Bulk Input

The UI exposes a CSV/TXT import dialog for active workflows. Templates are generated by the backend.

Create Class CSV headers:

```csv
class_name,academic_year_id,section_count,capacity,section_names,subjects,fees
PP1,academic-year-id,2,30,A|B,English|Math,Tuition=1000|Books=500
```

Create Class TXT format:

```txt
class_name:
academic_year_id:
section_count:
capacity:
section_names:
subjects:
fees:
```

Imported content is previewed into the workflow draft first. Execution still requires validation and confirmation.

## Mobile Preview

Open `docs/guided-assistant-mobile-preview.html` to review the intended mobile experience. It includes the assistant landing screen, create-class chat flow, rule-based suggestions, review/confirmation, and bulk CSV/TXT draft import.

## Compatibility Guardrails

- Existing ERP routes and screens are not removed.
- Existing CRUD endpoints remain available.
- Assistant execution uses transactions and rolls back on failure.
- Drafts can be resumed or canceled without touching ERP records.
- Rule-based validation is deterministic; no uncontrolled AI chat behavior is used.
